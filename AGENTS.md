# 주몽신화전기 Codex 운영 규칙

## 프로젝트 전제

- Godot 4.x 기반 2D SD 도트 캐릭터 중심의 쿼터뷰 턴제 RPG로 개발한다.
- 주요 범위는 주몽의 탄생, 동부여 성장과 갈등, 오이·마리·협보와의 탈출, 엄사수 도하, 졸본 정착과 고구려 건국이다.
- 캐릭터, 전술 맵, UI와 핵심 게임플레이는 2D 제작 파이프라인을 기준으로 구성하고 Steam 상업 출시와 상업적 성공을 목표로 한다.
- 신규 제작에는 `Node2D`, `TileMapLayer`, `Camera2D`, `CharacterBody2D`, `Area2D`, `Control`, `CanvasLayer` 등 Godot 2D 노드를 사용한다.
- 과거 HD-2D·3D 시험 씬과 전용 자산은 삭제했다. 이를 복원하지 않고 현재 정본에 맞는 자산만 새로 제작한다.
- 3D 기술 자체를 금지하지 않는다. 승인된 교전 컷인의 작은 배경 무대와 고립된 이펙트에는 사용할 수 있으나, 2D 캐릭터 정체성, 2D 폴백, 게임플레이 계산 비의존, 전용 경로, 화면·Steam Deck 성능 검증을 모두 갖춰야 한다.
- 원사료와 역사적 명칭을 우선하되 게임에 필요한 각색과 판타지를 허용한다.
- PC Steam 출시를 기준으로 키보드·마우스와 게임패드를 모두 지원하고, 한국어·영어를 우선하는 확장 가능한 다국어 구조를 사용한다.

## 감독 승인 게이트

- 모든 제작 요청은 먼저 `jumong-production-governor`가 2D 캐릭터 중심 정체성, 승인된 하이브리드 범위, 담당 에이전트와 파일 소유권 충돌을 점검한다.
- 제작·수정 요청을 받으면 먼저 읽기 전용으로 조사하고 실행 계획을 제출한다.
- 계획에는 목표, 담당 영역, 생성·수정 예정 파일, 영향, 위험, 검증 방법을 포함한다.
- 새 마일스톤, 작업 패키지 또는 아래 결정 게이트는 사용자의 명시적 승인을 받는다.
- 사용자가 한 번 승인한 목표·파일·완료 기준 안에서는 되돌릴 수 있는 비결정 구현, 버그 수정, 테스트와 문서 동기화를 자율적으로 계속한다.
- 사용자가 구체적인 파일 변경을 직접 명령한 경우 그 범위는 승인된 것으로 본다.
- 승인 범위를 임의로 확대하지 않는다. 광범위한 변경은 별도 승인을 받는다.
- 구현 결과를 스스로 최종 완료로 판정하지 않는다. `jumong-verify` 검증과 사용자 확인을 거친다.
- 검증된 승인 범위와 일일 보고서의 fast-forward 커밋·`origin/main` 푸시는 사용자의 상시 승인 범위다. force push, amend, reset, 자동 rebase와 이력 재작성은 금지한다.

## 결정 게이트와 자율 진행

다음은 반드시 일일 보고서에 선택지·추천·영향을 기록하고 사용자 결정을 받는다.

- 유료 구매, 구독, 후원, 외부 계약과 결제
- 무료를 포함한 새 플러그인·런타임 의존성의 프로젝트 도입
- GDScript에서 C#으로의 전환이나 C++ GDExtension 도입처럼 빌드·배포 체계를 바꾸는 기술 스택 변경
- 라이선스, 저작권, 상업 이용, Steam AI 콘텐츠 공개 판단
- 장르, 핵심 전투, 시각 정본, 서사 정본, 마일스톤 범위의 변경
- 호환성을 깨는 데이터 스키마 변경, 대량 폐기·이관, 되돌리기 어려운 변경
- Steam 가격, 출시일, 할인, 데모·Next Fest 시점, 상점 공개, 태그 정본, 마케팅 문구
- 외부 게시, 계정 로그인, 개인정보 입력과 공개 상태 변경

다음은 승인된 작업 범위 안에서 계속 진행한다.

- 확정된 사양의 구현과 되돌릴 수 있는 버그 수정·리팩터링
- 정적 타입, 시그널, 오류 처리, 테스트와 검증 준비
- 플레이스홀더, 읽기 전용 조사, 문서·데이터 동기화
- 완료 기준과 공개 계약을 바꾸지 않는 세부 구현 선택

## 역할 분류

`jumong-production-governor`가 방향과 역할 경계를 통제하고 `jumong-game-director`가 요청을 분류·통합한다.

- `jumong-production-governor`: 2D 캐릭터 중심 정체성, 하이브리드 승인 범위, 운영문서 일관성, 역할·파일 소유권 통제
- `jumong-game-director`: 승인 계획, 마일스톤 분류, 통합 순서
- `jumong-narrative`: 플롯, 퀘스트, 이벤트, 대사, 로어, 고증 용어
- `jumong-level-2d`: 2D 배경, 이동 경로, 조우, Camera2D, 컷신 무대
- `jumong-character`: 파티·NPC·적 설정, 능력치, 성장, 2D 액터 씬 통합
- `jumong-2d-art`: 스프라이트, 배경, 타일, UI 그래픽, 2D VFX와 애셋 기록
- `jumong-core-systems`: 부트스트랩, 장면 전환, 입력, 표시 설정, 세이브·로드 기반
- `jumong-ui-systems`: HUD, 메뉴, 대화, 결과 화면, 옵션과 저장 UI
- `jumong-combat`: 턴 상태머신, 스킬, 데미지, 상태이상, 적 AI, 승패
- `jumong-audio`: BGM, 효과음, 음성, 큐 데이터, 버스와 런타임 재생
- `jumong-steam-strategy`: 최신 Steam 정책·행사·상점·가격·현지화·Deck 전략 조사와 에이전트 협의
- `jumong-verify`: 실행, 화면·로그·오류 확인과 증거 기반 판정

여러 영역이 관련되면 책임과 파일 소유권을 계획에 분리한다. 동일 파일 변경은 순차 수행하고 통합 후 검증한다. 서브에이전트는 사용자가 명시적으로 요청한 경우에만 사용한다.

## 전투 프로토타입 공통 정본

- 전투 관련 작업을 시작하기 전에 `docs/production/combat_prototype_agent_brief_v1.md`를 반드시 읽는다.
- 전술 맵, 다수 병사 교전 컷인, 개인·서사 컷신을 서로 다른 연출 레이어로 유지한다.
- 분대를 지휘관 1명+병사 3명 또는 총 4명으로 축소 해석하지 않는다. M4 기본은 지휘관 1명+병사 8명이며 데이터상 병사 6~12명을 허용한다.
- 모든 전투 구현은 `tests/2d_identity_smoke.gd`를 통과해야 한다.
- 승인된 3D 미니 필드는 교전 컷인 배경으로만 격리하고, 전투 계산·충돌·물리·내비게이션을 소유하지 않으며 동일 payload의 2D 폴백을 유지한다.

## 애셋 제작·구매 정책

- 담당자는 직접 제작보다 구매·도입이 일정, 품질, 유지보수와 상업 리스크 측면에서 낫다고 판단하면 [Godot Asset Store](https://store.godotengine.org/)를 우선 조사해 구매 요청을 제출한다.
- 요청에는 이름, 링크, 가격, 라이선스, Godot 버전, 전체 2D 적합성, 제작 대비 절감, 화풍 적합성, 통합 위험, 대체 후보와 `제작/구매/보류` 추천을 포함한다.
- 구매, 결제, 로그인, 다운로드와 약관 수락은 사용자 승인 없이 수행하지 않는다.
- 무료 애셋도 플러그인·런타임 의존성 또는 독점 라이선스를 포함하면 결정 게이트를 적용한다.
- 음악·효과음이 필요하면 공식 라이선스를 확인할 수 있는 무료 음원 사이트도 조사한다. 사이트가 아니라 파일별로 상업 이용, 수정, 재배포, 출처표기, 생성형 AI·Content ID 여부를 기록한다.
- CC0이며 권리 출처가 명확한 파일은 승인 범위 안에서 도입할 수 있다. CC BY, 사이트 자체 라이선스, 로그인 필요, 독점 조항, AI 생성, 음성, Content ID 가능성이 있는 파일은 일일 보고의 결정 항목으로 올린다.
- 비상업·개인용·편집용 전용, 변경 금지, 출처 불명, 게임·영상 추출 음원은 프로젝트에 포함하지 않는다.

## Steam 시장 협의

- `jumong-steam-strategy`는 주 1회와 각 마일스톤 시작 시 Steamworks 공식 자료를 우선해 최신 동향을 조사한다.
- 모든 전문 에이전트는 최신 브리프를 읽고 자기 영역의 적용 가치, 비용, 위험, 보류 이유를 제출한다.
- `jumong-game-director`가 공통안과 이견을 통합하고 `jumong-production-governor`가 전체 2D·범위 적합성을 감사한다.
- 조사 결과는 근거이며 자동 제작 명령이 아니다. 핵심 방향 변경만 사용자에게 최종 결정 요청으로 올린다.

## 구현 규칙

- 프로젝트는 각 승인 단계 뒤에도 실행 가능한 상태를 유지한다.
- 신규 게임플레이, 전술 카메라, 충돌, 전술 맵과 캐릭터 구현은 2D 노드와 2D 애셋을 기준으로 한다. 승인된 교전 컷인 배경과 고립된 하이브리드 이펙트는 전용 경로, 2D 폴백, 계산 비의존, 성능·화면 검증을 유지한다.
- GDScript에는 가능한 한 정적 타입 힌트를 사용한다.
- 기본 구현 언어는 GDScript로 유지한다. C#은 검증된 라이브러리·도구 이점이 빌드 복잡도보다 클 때, C++은 성능 프로파일이나 Steam SDK 네이티브 연동이 요구할 때만 결정 게이트를 거쳐 도입한다.
- 사용자에게 보이는 문자열은 코드와 씬에 새로 하드코딩하지 않고 번역 키·외부 데이터로 관리한다. 한국어를 원문 정본으로 두되 영어 UI 길이와 폰트 폴백을 함께 검증한다.
- 데이터와 로직을 분리하고 스킬 수치, 대사, 이벤트는 적절한 `.tres`, JSON 또는 별도 리소스로 관리한다.
- 전투 흐름은 명시적 상태머신으로 구성한다.
- UI와 전투 로직은 시그널과 이벤트로 연결한다.
- 동일 파일의 주 소유자는 한 에이전트로 지정하며 협업 변경은 순차 수행한다.
- 씬·리소스 편집 뒤 Godot 파일시스템 재탐색 또는 헤드리스 에디터 로드를 수행한다.
- 프로토타입은 플레이스홀더를 우선하고 특정 게임·드라마·영화 디자인을 복제하지 않는다.
- 실제 포함되는 AI 생성 애셋·대사·로어는 생성 도구, 날짜, 목적, 수정 여부, 사용 위치, 저작권 검토를 기록한다.
- Steam 제출 전 AI 콘텐츠 공개 정책은 최신 공식 자료로 다시 확인한다.

## 검증 규칙

- 구현 뒤 `jumong-verify`를 사용한다.
- 검증 중 코드, 씬, 리소스, 애셋을 수정하지 않는다.
- Windows의 제한된 Codex 샌드박스에서 Godot 실행 파일을 직접 시작하지 않는다. 해당 환경에서는 Godot 4.7.1이 네이티브 예외로 종료되어 Visual Studio JIT 디버거 창을 만들 수 있다.
- Godot 검증은 총괄 또는 `jumong-verify` 한 곳에서 `tools/run_godot_check.ps1`로만 순차 실행한다. 샌드박스 거부 코드 `86`이 나오면 같은 명령을 정상 사용자 환경 권한으로 다시 실행한다.
- 여러 에이전트가 Godot 편집기·테스트를 병렬 실행하지 않는다. 실행 잠금 또는 기존 Godot 프로세스 코드 `87`이 나오면 현재 검증이 끝날 때까지 기다린다.
- 충돌을 숨기기 위해 Windows JIT 디버깅을 전역 비활성화하지 않는다. 원인 명령, 종료 코드와 프로세스 잔류 여부를 기록한다.
- 변경 스크립트·씬은 Godot 헤드리스 로드 또는 관련 씬 실행으로 파싱·런타임 오류를 확인한다.
- 화면이나 로그로 확인되지 않은 기능은 통과시키지 않는다.
- 결과는 `통과`, `부분 통과`, `실패`, `검증 불가`로 보고하고 명령, 종료 코드, 오류·경고, 증거, 미확인 항목을 명시한다.
- 검증되지 않은 작업은 완료로 표현하지 않는다.
