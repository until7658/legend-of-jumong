# 주몽신화전기 Codex 운영 규칙

## 프로젝트 전제

- Godot 4.x 기반 HD-2D 턴제 RPG로 개발한다.
- 주요 범위는 주몽의 탄생, 동부여 성장과 갈등, 오이·마리·협보와의 탈출, 엄사수 도하, 졸본 정착과 고구려 건국이다.
- 2D 캐릭터와 3D 배경을 조합하고 Steam 상업 배포를 고려한다.
- 원사료와 역사적 명칭을 우선하되 게임에 필요한 각색과 판타지를 허용한다.

## 감독 승인 게이트

- 제작·수정 요청을 받으면 먼저 읽기 전용으로 조사하고 실행 계획을 제출한다.
- 계획에는 목표, 담당 영역, 생성·수정 예정 파일, 영향, 위험, 검증 방법을 포함한다.
- 사용자의 명시적 승인 전에는 프로젝트 파일이나 외부 상태를 변경하지 않는다.
- 사용자가 구체적인 파일 변경을 직접 명령한 경우 그 범위는 승인된 것으로 본다.
- 승인 범위를 임의로 확대하지 않는다. 광범위한 변경은 별도 승인을 받는다.
- 구현 결과를 스스로 최종 완료로 판정하지 않는다. `jumong-verify` 검증과 사용자 확인을 거친다.
- 체크포인트나 커밋은 사용자가 명시적으로 요청하거나 승인한 경우에만 생성한다.

## 역할 분류

`jumong-game-director`로 요청을 분류하고 통합한다.

- narrative: 플롯, 퀘스트, 이벤트, 대사, 로어, 고증 용어
- character: 파티·NPC·적 설정, 능력치, 성장, 스프라이트 사양, 씬 통합
- skill-combat: 턴 상태머신, 스킬, 데미지, 상태이상, 승패
- level-map: 3D 배경, 이동 경로, 조우, 컷신 카메라, HD-2D 연출
- asset: 스프라이트, 텍스처, UI 그래픽, 이펙트, 셰이더, 오디오
- ui-systems: HUD, 메뉴, 결과 화면, 세이브·로드, 옵션
- verifier: 실행, 화면·로그·오류 확인과 증거 기반 판정

여러 영역이 관련되면 책임과 파일 소유권을 계획에 분리한다. 동일 파일 변경은 순차 수행하고 통합 후 검증한다. 서브에이전트는 사용자가 명시적으로 요청한 경우에만 사용한다.

## 구현 규칙

- 프로젝트는 각 승인 단계 뒤에도 실행 가능한 상태를 유지한다.
- GDScript에는 가능한 한 정적 타입 힌트를 사용한다.
- 데이터와 로직을 분리하고 스킬 수치, 대사, 이벤트는 적절한 `.tres`, JSON 또는 별도 리소스로 관리한다.
- 전투 흐름은 명시적 상태머신으로 구성한다.
- UI와 전투 로직은 시그널과 이벤트로 연결한다.
- 씬·리소스 편집 뒤 Godot 파일시스템 재탐색 또는 헤드리스 에디터 로드를 수행한다.
- 프로토타입은 플레이스홀더를 우선하고 특정 게임·드라마·영화 디자인을 복제하지 않는다.
- 실제 포함되는 AI 생성 애셋·대사·로어는 생성 도구, 날짜, 목적, 수정 여부, 사용 위치, 저작권 검토를 기록한다.
- Steam 제출 전 AI 콘텐츠 공개 정책은 최신 공식 자료로 다시 확인한다.

## 검증 규칙

- 구현 뒤 `jumong-verify`를 사용한다.
- 검증 중 코드, 씬, 리소스, 애셋을 수정하지 않는다.
- 변경 스크립트·씬은 Godot 헤드리스 로드 또는 관련 씬 실행으로 파싱·런타임 오류를 확인한다.
- 화면이나 로그로 확인되지 않은 기능은 통과시키지 않는다.
- 결과는 `통과`, `부분 통과`, `실패`, `검증 불가`로 보고하고 명령, 종료 코드, 오류·경고, 증거, 미확인 항목을 명시한다.
- 검증되지 않은 작업은 완료로 표현하지 않는다.

## 마일스톤 종료 보고

- 승인받은 범위와 변경 파일
- 실행·검증 대상과 명령
- 화면 확인 여부와 증거 경로
- 오류 및 경고와 검증 판정
- 남은 작업과 재검증 항목
