# 주몽신화전기

## 에이전트 팀 구성 및 운영 가이드

## 1. 문서 목적

본 문서는 Godot 4.x를 사용해 **주몽 신화 기반 HD-2D 턴제 RPG**를 개발할 때 필요한 제작 역할을 서브에이전트로 구성하고 운영하는 기준을 정의한다.

시나리오, 캐릭터, 전투, 맵, 애셋, UI, 검증 등의 작업은 각 담당 에이전트가 수행하며, 사용자는 프로젝트 감독으로서 다음 역할을 담당한다.

* 작업 목표와 마일스톤 지시
* 에이전트가 제출한 계획 검토
* 계획 승인, 수정 또는 반려
* 실행 결과와 검증 증거 확인
* 다음 작업 진행 여부 결정

모든 제작 작업은 **사용자 승인 전에는 실행하지 않는 것**을 기본 원칙으로 한다.

---

## 2. 프로젝트 기본 전제

### 2.1 기술 및 장르

* 엔진: Godot 4.x
* 라이선스: MIT
* 장르: HD-2D 턴제 RPG
* 표현 방식:

  * 캐릭터: 2D 스프라이트
  * 배경 및 전투 공간: 3D
  * 이펙트: 2D/3D 파티클 및 셰이더 혼합
* 배포 플랫폼: Steam
* 배포 형태: 상업 판매

### 2.2 주요 스토리 범위

주몽 신화의 주요 사건을 중심으로 구성한다.

1. 주몽의 탄생
2. 동부여에서의 성장
3. 대소 왕자들과의 갈등
4. 오이·마리·협보와 함께 동부여 탈출
5. 엄사수 도하
6. 졸본 정착과 고구려 건국

원사료와 역사적 명칭을 기반으로 하되, 게임으로서 필요한 각색과 판타지 요소를 추가할 수 있다.

## 3. 핵심 운영 원칙

### 3.1 감독 컨펌 모델

모든 제작 에이전트는 다음 절차를 따른다.

```text
사용자 작업 지시
    ↓
담당 에이전트가 실행 계획 제출
    ↓
사용자 검토
    ├─ 승인
    ├─ 수정 지시
    └─ 반려
    ↓
승인된 범위만 실행
    ↓
verifier가 실행 및 검증
    ↓
스크린샷·로그·에러 보고
    ↓
사용자 최종 확인
    ├─ 통과 및 체크포인트 생성
    └─ 재작업 지시
```

### 3.2 공통 강제 규칙

모든 제작 에이전트는 다음 규칙을 준수한다.

* 실행 전에 반드시 계획을 제출한다.
* 사용자 승인 없이 파일을 생성하거나 수정하지 않는다.
* 승인받은 범위를 임의로 확대하지 않는다.
* 스스로 작업 완료를 선언하지 않는다.
* 완료 여부는 verifier 검증 결과로 판단한다.
* 작업 후에도 프로젝트가 실행 가능한 상태를 유지한다.
* 광범위하거나 위험한 변경은 별도의 승인을 받는다.
* 마일스톤 완료 시 체크포인트를 생성한다.

### 3.3 권한 운영

* 기본 권한 모드: `plan`
* 자동 편집 및 자동 실행 금지
* 전체 권한 우회 또는 blanket permission 허용 금지
* 검증된 단순 반복 작업만 제한적으로 자동화
* 반려된 변경은 체크포인트를 통해 즉시 되돌릴 수 있어야 한다

## 4. 에이전트 팀 구성

```text
감독 사용자
    │
    │ 지시 / 승인 / 반려
    ▼
오케스트레이터
    ├─ narrative
    ├─ character
    ├─ skill-combat
    ├─ level-map
    ├─ asset
    ├─ ui-systems
    └─ verifier
           │
           ▼
    Godot MCP 서버
           │
           ▼
    Godot 에디터 및 실행 게임
```

| 에이전트         | 주요 역할                |     모델 | 권한    |
| ------------ | -------------------- | -----: | ----- |
| narrative    | 시나리오, 퀘스트, 이벤트, 대사   |   opus | plan  |
| character    | 캐릭터 설정, 스탯, 스프라이트 사양 | sonnet | plan  |
| skill-combat | 턴제 전투, 스킬, 데미지, 상태이상 |   opus | plan  |
| level-map    | 3D 배경, 레벨 구성, 조우 배치  | sonnet | plan  |
| asset        | 스프라이트, 텍스처, 오디오, 이펙트 | sonnet | plan  |
| ui-systems   | HUD, 메뉴, 세이브 및 로드    | sonnet | plan  |
| verifier     | 실행, 스크린샷, 로그 검증      |  haiku | 읽기 전용 |

에이전트 정의 파일은 다음 위치에 배치한다.

```text
.claude/
├─ agents/
│  ├─ narrative.md
│  ├─ character.md
│  ├─ skill-combat.md
│  ├─ level-map.md
│  ├─ asset.md
│  ├─ ui-systems.md
│  └─ verifier.md
├─ commands/
│  ├─ plan.md
│  └─ verify.md
└─ settings.json
```

## 5. 에이전트별 역할

## 5.1 narrative

### 담당 업무

* 전체 스토리의 3막 또는 4막 구조 설계
* 메인 이벤트와 서브 퀘스트 구성
* 컷신, 대화, 로어 작성
* 분기 조건 및 플래그 정의
* 고증 용어와 인물 관계 검토

### 주요 스토리 구조

* 탄생과 동부여 성장
* 대소 왕자들과의 갈등
* 동부여 탈출
* 엄사수 도하
* 졸본 정착과 고구려 건국

### 작업 규칙

* 대사와 이벤트는 코드에 직접 작성하지 않는다.
* `.tres`, JSON 또는 별도 리소스로 분리한다.
* 작업 전 플롯과 이벤트 목록을 먼저 제출한다.
* 게임에 포함되는 AI 생성 대사와 로어는 AI 생성 대장에 기록한다.
* 엄사수, 졸본, 오이, 마리, 협보 등의 명칭을 일관되게 사용한다.

---

## 5.2 character

### 담당 업무

* 플레이어 파티 구성
* 주요 NPC와 적 캐릭터 정의
* 역할, 능력치, 성장 구조 설계
* 캐릭터별 스프라이트 요구사항 작성
* 완성된 애셋을 캐릭터 씬에 통합

### 기본 파티 예시

* 주몽: 원거리 공격, 궁술, 빛 또는 태양 속성
* 오이: 전방 공격 또는 방어 역할
* 마리: 기동형 공격 또는 보조 역할
* 협보: 지원, 약화 또는 전술 역할

### 스프라이트 사양 예시

* 대기
* 이동
* 일반 공격
* 피격
* 스킬
* 승리
* 전투 불능

실제 스프라이트 생성은 asset 에이전트에 요청하고, character 에이전트는 요구사항 정의와 씬 통합을 담당한다.

---

## 5.3 skill-combat

### 담당 업무

* 턴 순서 계산
* 행동 선택
* 공격 및 스킬 처리
* 데미지 계산
* 상태이상 적용과 해제
* 승리 및 패배 판정
* 전투 상태머신 구현

### 구현 원칙

* 전투 흐름은 명확한 상태머신으로 구성한다.
* 스킬 수치는 `.tres` 리소스로 분리한다.
* GDScript 정적 타입 힌트를 사용한다.
* UI 연결은 직접 참조보다 시그널을 우선한다.
* 구현 전에 상태 다이어그램과 데미지 공식을 제출한다.
* 구현 후 verifier에게 검증을 요청한다.

### 상태 예시

```text
BattleStart
→ TurnStart
→ CommandSelect
→ TargetSelect
→ ActionExecute
→ StatusResolve
→ VictoryCheck
→ TurnEnd
→ 다음 TurnStart
```

---

## 5.4 level-map

### 담당 업무

* 동부여 궁정
* 동부여 외곽 및 탈출로
* 엄사수 강가
* 졸본 지역
* 필드 이동 경로
* 전투 조우 지점
* 컷신 카메라 위치

### HD-2D 표현 원칙

* 2D 스프라이트를 3D 공간에 배치한다.
* 카메라는 원근감을 제한적으로 사용한다.
* 얕은 피사계심도와 블룸을 적용한다.
* 캐릭터 가독성을 해치지 않는 조명을 사용한다.
* 배경 오브젝트의 크기와 깊이를 통해 입체감을 표현한다.

씬이나 리소스를 편집한 뒤에는 Godot 파일시스템 재탐색을 수행한다.

---

## 5.5 asset

### 담당 업무

* 캐릭터 스프라이트
* 배경 텍스처
* UI 그래픽
* 파티클 이펙트
* 셰이더
* 효과음
* 배경 음악
* 프로젝트 임포트와 애셋 연결

### 제작 구분

#### 에이전트가 직접 제작 가능한 항목

* 도형 기반 플레이스홀더
* 간단한 UI 그래픽
* 파티클 이펙트
* 셰이더
* 프로시저럴 텍스처
* 임시 효과음 또는 테스트용 리소스

#### 별도 생성 백엔드가 필요한 항목

* 픽셀아트 캐릭터
* 일러스트
* 배경 원화
* 음성
* 배경 음악
* 고품질 효과음

프로토타입 단계에서는 플레이스홀더를 우선 사용하고, 실제 생성형 미디어 애셋은 별도의 마일스톤으로 진행한다.

### 저작권 및 AI 생성 관리

* 타 게임, 드라마, 영화의 디자인을 복제하지 않는다.
* 원사료, 유물, 유적, 전통 복식 자료를 참고한다.
* 생성 결과가 특정 저작물을 모방하지 않았는지 확인한다.
* 게임에 포함되는 AI 생성 애셋은 대장에 기록한다.
* 최종 아트 디렉션은 사용자가 승인한다.

---

## 5.6 ui-systems

### 담당 업무

* 전투 HUD
* 파티 HP 및 MP 표시
* 턴 순서 표시
* 행동 선택 메뉴
* 스킬 및 아이템 메뉴
* 메인 메뉴
* 일시정지 메뉴
* 결과 화면
* 세이브 및 로드
* 옵션 설정

### 구현 원칙

* 전투 로직과 UI를 직접 결합하지 않는다.
* 시그널과 이벤트를 통해 상태를 전달한다.
* UI는 전투 데이터를 표시하고 명령을 전달하는 역할만 담당한다.
* 작업 전 화면 구성과 사용자 흐름을 제출한다.

---

## 5.7 verifier

### 담당 업무

* 게임 또는 지정 씬 실행
* 스크린샷 캡처
* 디버그 로그 수집
* 에러와 경고 확인
* 씬 트리 확인
* 요청한 기능이 실제로 동작하는지 판정

### 검증 원칙

* 코드, 씬, 애셋을 직접 수정하지 않는다.
* 화면이나 로그에서 확인되지 않은 기능은 통과시키지 않는다.
* 추측에 의한 성공 판정을 금지한다.
* 검증 실패 시 `미반영`, `부분 반영`, `오류 발생`으로 명확히 보고한다.

### 검증 보고 형식

```text
검증 대상:
실행 씬:

판정:
- 통과
- 부분 통과
- 실패

확인 결과:
- 요청 동작
- 화면 확인 내용
- 로그 확인 내용

오류 및 경고:
- 오류 개수
- 주요 메시지

증거:
- 스크린샷
- 실행 로그
- 씬 트리

다음 조치:
- 수정 필요 사항
- 재검증 항목
```

## 6. 프로젝트 공통 규칙

프로젝트 루트에 `CLAUDE.md`를 배치한다.

```markdown
# 주몽신화전기 프로젝트 규칙

## 프로젝트

- Godot 4.x
- HD-2D 턴제 RPG
- 주몽 신화 기반
- Steam 상업 배포
- 2D 캐릭터와 3D 배경 조합

## 감독 승인 원칙

- 제작 에이전트는 실행 전 계획을 제출한다.
- 사용자 승인 후에만 변경을 실행한다.
- 에이전트는 스스로 완료를 선언하지 않는다.
- 완료 판정은 verifier의 실행, 스크린샷, 로그 검증으로 결정한다.
- 프로젝트는 항상 실행 가능한 상태를 유지한다.

## 코드 및 씬

- GDScript 정적 타입 힌트를 사용한다.
- 데이터와 로직을 분리한다.
- UI와 로직은 시그널 기반으로 연결한다.
- 씬 편집 후 파일시스템을 다시 탐색한다.
- 광범위한 변경은 별도 승인을 받는다.

## 애셋 및 저작권

- 프로토타입은 플레이스홀더를 사용한다.
- AI 생성 애셋은 AI 생성 대장에 기록한다.
- 생성 애셋은 오리지널성과 비침해 여부를 검토한다.
- 특정 드라마, 게임, 영화의 디자인을 복제하지 않는다.
- 원사료와 역사 자료를 우선 참고한다.
```

## 7. AI 생성 콘텐츠 관리

게임에 실제로 포함되는 AI 생성 콘텐츠는 별도 대장으로 관리한다.

### 관리 대상

* 캐릭터 및 몬스터 스프라이트
* 배경 이미지와 텍스처
* UI 이미지
* 음성
* 음악과 효과음
* 대사
* 로어
* 컷신 이미지

### 대장 필드 예시

| 필드          | 내용               |
| ----------- | ---------------- |
| Asset ID    | 고유 식별자           |
| 파일 경로       | 프로젝트 내 경로        |
| 애셋 종류       | 이미지, 음성, 음악, 텍스트 |
| 생성 도구       | 사용한 생성 도구        |
| 생성 일자       | 생성 날짜            |
| 프롬프트 요약     | 제작 목적과 주요 지시     |
| 수정 여부       | 수작업 또는 후처리 여부    |
| 사용 위치       | 씬, 캐릭터, 이벤트      |
| 저작권 검토      | 오리지널 및 비침해 확인    |
| Steam 공개 대상 | 예 또는 아니오         |

코드, 씬 구조, 데이터 파일과 같은 개발 산출물과 플레이어에게 노출되는 생성 콘텐츠를 구분해 관리한다.

Steam 제출 전에는 최신 AI 콘텐츠 공개 정책과 콘텐츠 설문 항목을 다시 확인한다.

## 8. 훅 및 자동 검증

## 8.1 PostToolUse

스크립트나 씬이 변경된 뒤 다음 작업을 수행한다.

1. Godot 헤드리스 스모크 테스트
2. 파싱 오류 확인
3. 런타임 오류 확인
4. 디버거 로그 수집
5. 실패 시 즉시 보고

## 8.2 Stop 및 SubagentStop

에이전트 종료 시 다음 사항을 요약한다.

* 변경한 파일
* 승인받은 작업 범위
* 실행 여부
* 스크린샷 확인 여부
* 오류 존재 여부
* verifier 검증 여부
* 남은 작업

검증되지 않은 작업은 완료로 표시하지 않고 verifier 호출을 다음 작업으로 안내한다.

## 9. 슬래시 커맨드

반복 작업은 `.claude/commands/`에 정의한다.

### `/plan`

* 현재 요청을 분석한다.
* 수정 예정 파일과 씬을 제시한다.
* 변경 이유와 영향 범위를 설명한다.
* 위험 요소와 검증 방법을 제시한다.
* 실행하지 않고 사용자 승인을 기다린다.

### `/verify`

* 현재 씬 또는 지정 씬을 실행한다.
* 스크린샷을 캡처한다.
* 디버그 출력과 오류를 수집한다.
* 요청 동작의 반영 여부를 판정한다.
* 코드나 씬은 수정하지 않는다.

## 10. 세션 운영 방법

### 10.1 세션 시작

1. Godot 에디터 실행
2. Godot MCP 플러그인 활성화
3. 프로젝트 폴더에서 Claude Code 실행
4. 권한 모드가 `Manual` 또는 `plan`인지 확인
5. MCP 서버 연결 상태 확인
6. 현재 체크포인트와 작업 브랜치 확인

```bash
claude
```

```text
/permissions
```

### 10.2 에이전트 호출

#### 자동 위임

사용자 요청이 에이전트 설명과 일치하면 오케스트레이터가 담당 에이전트에 위임한다.

```text
전투 턴 순서 상태머신을 설계해줘.
```

→ `skill-combat`

#### 명시적 호출

```text
@narrative 1막 동부여 구간의 플롯을 계획해줘.
```

```text
@level-map 엄사수 강가 씬 레이아웃을 제안해줘.
```

```text
@verifier 현재 전투 씬을 실행하고 검증해줘.
```

### 10.3 병렬 작업

서브에이전트는 병렬로 실행할 수 있지만, 감독 승인 모델에서는 다음 원칙을 권장한다.

* 계획 수립은 병렬 가능
* 동일 파일을 변경하는 작업은 순차 실행
* 마일스톤 구현은 담당 에이전트별로 분리
* 검증은 통합 후 수행
* 승인되지 않은 계획끼리 자동 병합하지 않음

## 11. 감독 컨펌 사이클 예시

### 사용자 지시

```text
@skill-combat M3 전투 골격을 만들어줘.
실행하지 말고 상태 다이어그램, 파일 목록, 데미지 공식부터 제안해줘.
```

### 에이전트 계획

* 전투 상태 다이어그램
* 생성 및 수정 파일 목록
* 턴 순서 계산 방식
* 기본 데미지 공식
* UI 전달 시그널
* 테스트 방법
* verifier 검증 항목

### 사용자 승인

```text
기본 계획은 승인한다.
다만 데미지 공식에 방어 관통을 추가하고, 랜덤 편차는 ±5%로 제한해줘.
```

### 구현 및 검증

1. 승인된 내용만 구현
2. verifier가 전투 씬 실행
3. 턴 순환 로그 확인
4. 행동 메뉴 스크린샷 확인
5. 오류 및 경고 확인
6. 사용자 최종 승인
7. 체크포인트 생성

## 14. 권장 시작 순서

가장 먼저 M1 환경 구축을 완료한다.

1. `CLAUDE.md` 작성
2. 에이전트 7종 정의
3. `/plan`, `/verify` 커맨드 작성
4. 검증 훅 설정
5. Godot MCP 연결
6. 빈 씬 실행 검증
7. 체크포인트 생성

M1 검증이 완료된 뒤 M2 HD-2D 화면 구축으로 진행한다.

전투 시스템인 M3부터 먼저 구현할 수도 있지만, MCP 연결과 verifier 동작을 검증하지 않은 상태에서 제작을 시작하면 이후 결과 검증과 되감기가 어려워질 수 있다.
