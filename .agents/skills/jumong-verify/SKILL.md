# 주몽 검증

## 읽기 전용 경계

- 코드, 씬, 리소스, 애셋을 생성·수정·삭제하지 않는다.
- 검증 도구가 만드는 Godot 캐시와 로그 외에는 프로젝트 상태를 변경하지 않는다.
- 실패를 고치지 말고 재현 조건과 수정 필요 사항만 보고한다.
- 검증 때문에 임포트 파일이나 프로젝트 파일이 바뀌면 그 사실을 판정과 분리해 보고한다.

## 검증 절차

1. `AGENTS.md`, `$jumong-production-governor`의 2D 캐릭터 중심 정본과 하이브리드 승인 범위, 승인 기준과 변경 파일 목록을 확인한다.
2. 사용 가능한 Godot 실행 파일과 프로젝트 메인 씬을 읽기 전용으로 확인한다.
3. 위험이 낮은 검사부터 수행한다.
   - 게임플레이·맵·카메라·충돌·캐릭터 경로가 2D인지 확인
   - 승인된 하이브리드 컷인·이펙트가 있다면 전용 경로, 최종 2D 합성, 게임플레이 비의존, 2D 폴백과 성능 증거를 확인
   - 프로젝트 헤드리스 에디터 로드
   - 지정 씬 또는 프로젝트 헤드리스 실행
   - 필요한 경우 화면 실행과 스크린샷 캡처
4. Windows에서는 Godot 실행을 `tools/run_godot_check.ps1`에 단일 소유자로 위임하고 모든 테스트를 순차 실행한다.
   - 제한된 `CodexSandboxOffline` 계정에서는 Godot을 시작하지 않는다. runner 종료 코드 `86`을 확인하고 정상 사용자 환경 권한으로 재실행한다.
   - runner 종료 코드 `87`은 기존 Godot 프로세스 또는 검증 잠금이 있다는 뜻이다. 병렬 실행하지 말고 현재 실행이 끝난 뒤 재시도한다.
   - 서브에이전트는 Godot을 직접 실행하지 않고 구현 완료와 필요한 검증 명령만 총괄에 전달한다.
5. 명령, Godot 버전, 실행 대상, 종료 코드를 기록한다.
6. 파싱 오류, 런타임 오류, 경고와 디버그 출력 및 종료 후 Godot 프로세스 잔류 여부를 확인한다.
7. 승인 기준마다 관찰 가능한 증거를 연결한다.
8. 아래 판정 규칙과 보고 형식으로 결과를 제출한다.

GUI 실행이나 외부 도구 권한이 필요하면 승인을 요청한다. 화면을 확인할 수 없다면 로그 성공을 화면 성공으로 대신하지 말고 `검증 불가` 또는 `부분 통과`로 판정한다.

## 판정 규칙

- `통과`: 모든 승인 기준이 적절한 증거로 확인되고 차단 오류가 없다.
- `부분 통과`: 일부 기준만 확인되었거나 비차단 경고·미확인 항목이 남는다.
- `실패`: 오류가 재현되거나 승인 기준과 다른 동작이 관찰된다.
- `검증 불가`: 실행 환경, 권한, 의존성 또는 증거 부족으로 판정할 수 없다.

종료 코드 0만으로 기능을 통과시키지 않는다. 오류가 없다는 사실과 요청 기능이 동작한다는 사실을 별도로 검증한다.

## 보고 형식

```text
검증 대상:
승인 기준:
실행 환경:
실행 명령 및 종료 코드:

판정: 통과 | 부분 통과 | 실패 | 검증 불가

기준별 확인 결과:
- 기준:
  관찰:
  증거:

오류 및 경고:
- 개수와 주요 메시지

증거:
- 스크린샷 경로
- 로그 또는 관련 출력
- 씬 트리·상태 정보

미확인 항목:
다음 조치 및 재검증 항목:
```

증거가 없으면 `확보하지 못함`과 이유를 적는다. 검증되지 않은 항목을 완료라고 표현하지 않는다.
