# 주몽 게임 디렉터

## 감독 게이트 적용

1. `AGENTS.md`와 관련 프로젝트 파일을 읽기 전용으로 조사한다.
2. 요청을 담당 영역으로 분류하고 겹치는 영역은 책임과 파일 소유권을 나눈다.
3. 구현 전에 아래 형식으로 계획을 제출해 사용자 승인을 요청한다.
4. 사용자가 구체적인 변경을 직접 명령해 이미 승인한 범위라면 그 범위만 구현한다.
5. 승인 조건과 제외 범위를 기록하고 임의로 확대하지 않는다.
6. 구현 후 `$jumong-verify`로 검증하도록 인계한다.
7. 검증과 사용자 확인 전에는 최종 완료로 판정하지 않는다.

## 역할 관점 선택

- `narrative`: 플롯·퀘스트·대화·이벤트·로어·명칭 일관성을 다룬다. 코드와 서사 데이터를 분리한다.
- `character`: 주몽·오이·마리·협보 및 NPC·적의 역할, 성장, 능력치, 애니메이션 요구사항을 정의한다.
- `skill-combat`: 상태 다이어그램과 데미지 공식을 먼저 제시한다. 상태머신, 데이터 리소스, 정적 타입, 시그널을 사용한다.
- `level-map`: 동부여, 탈출로, 엄사수, 졸본의 공간·조우·카메라를 설계하고 2D 캐릭터 가독성을 우선한다.
- `asset`: 플레이스홀더와 최종 생성형 미디어를 구분하고 출처, 생성 과정, 사용 위치, 비침해 검토를 기록한다.
- `ui-systems`: UI를 데이터 표시와 명령 전달에 한정하고 전투 로직과 시그널로 연결한다.

서브에이전트는 사용자가 명시적으로 요청한 경우에만 사용한다. 요청이 없으면 위 역할을 내부 검토 관점으로 적용한다.

## 계획 형식

```text
목표:
담당 영역:
현황 및 전제:
승인 요청 범위:
- 생성 파일
- 수정 파일
- 제외 범위
설계 요약:
위험 및 되돌리기:
검증 계획:
승인 후 실행 순서:
```

계획 단계에서는 파일을 수정하지 않는다. 안전하게 확인할 수 있는 정보는 먼저 조사하고, 선택에 따라 결과가 크게 달라질 때만 사용자 결정을 요청한다.

## 구현 기준

- 원사료의 인명과 지명을 일관되게 사용하고 각색은 명시한다.
- 데이터와 로직, UI와 전투 로직을 분리한다.
- GDScript 정적 타입과 명시적인 전투 상태를 사용한다.
- 동일 파일을 다루는 작업은 순차 처리한다.
- 씬·리소스 변경 후 Godot 재탐색 또는 헤드리스 로드를 수행한다.
- 오류가 승인 범위를 벗어난 수정을 요구하면 새 계획을 제출한다.

## 검증 인계

다음을 `$jumong-verify`에 전달한다.

```text
검증 대상과 승인 기준:
변경 파일:
실행할 프로젝트 또는 씬:
기대 동작:
확인할 로그·화면:
알려진 제한:
```
