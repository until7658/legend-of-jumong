## M4 오전 추가 보고 · SD 도트 쿼터뷰 분대전

### 사용자 확정 방향

- 인게임 캐릭터는 2~4등신, 목표 약 3등신의 도트 캐릭터로 제작한다.
- 필드와 전술 맵은 45~50도 하향 쿼터뷰를 유지한다.
- 전투는 지휘관이 다수 병사를 편성·지휘하는 분대형 턴제 전술 전투로 전환한다.
- 분대는 총 4명이 아니다. 데이터상 지휘관 1명+병사 6~12명 가변이며 M4는 지휘관 1명+병사 8명을 표시한다.
- 전술 맵, 다수 병사 교전 컷인, 궁술대회·대화·결투 같은 개인 컷신을 서로 다른 연출 레이어로 구성한다.

### 에이전트 협의 결과

- production/game direction: 특정 전술 RPG의 UI·수치·구도는 복제하지 않고 `군령`, `대열`, `기세`를 프로젝트 고유 규칙으로 사용한다.
- combat: 궁병은 2칸 이상에서 창병에 유리하고 창병은 인접한 궁병에 유리한 위치 상성을 M4에서 먼저 검증한다.
- character/2d-art: 전술 맵은 8인 축약 대형, 교전 컷인은 양측 16명 규모 블록아웃, 개인 컷신은 주몽의 자세·호흡·활·화살 궤적 중심을 추천한다.
- ui/core: 1280×720 기준 viewport 정수 배율, Nearest 필터, 픽셀 스냅, 키보드·게임패드 공통 행동을 적용한다.
- audio/Steam: 전술 명령음·교전 충격음·서사 감정음을 분리하고 교전 컷인은 짧고 건너뛸 수 있게 설계한다.

### 구현

- `SquadBattleController` 명시적 상태머신: 선택→이동→군령→해결→적 턴→라운드 종료→승패
- 주몽+수련 궁병 8명 대 훈련대장+수련 창병 8명
- 이동, 기본 사격, 대열 방어, 턴 종료, 거리/인접 상성, 병사 기세와 지휘관 기세, 비살상 제압
- 타이틀→약 90초 프롤로그→분대 훈련전→결과→`m4_training_complete` 체크포인트 저장 흐름
- 런타임 입력 프로필 8개 행동과 버전 1 JSON 세이브 서비스
- 약 3등신 주몽 남향 도트 후보를 생성하고 투명화했으며 AI 애셋 대장에 `approval_candidate`로 기록
- 이전 6등신 비픽셀 신규 후보는 방향 폐기로 프로젝트에서 제거

## 오후 정정 · 2D 캐릭터 중심 정본과 교전 컷인 A/B 스파이크

### 사용자 승인과 운영 정정

- “3D 기술 금지”를 프로젝트 정체성으로 사용하지 않는다.
- 정본은 **2D SD 도트 캐릭터 중심의 쿼터뷰 턴제 RPG**다.
- 전술 맵·캐릭터·UI·조작·충돌·전투 계산은 2D를 기준으로 유지한다.
- 교전 컷인의 아주 작은 배경 무대에는 승인된 범위에서 3D를 사용할 수 있다.
- 승인 조건은 전용 경로, 전투 계산 비의존, 2D 캐릭터·VFX·UI 합성, 동일 payload의 2D 폴백, 화면·Steam Deck 성능 검증이다.
- 과거 HD-2D/3D 시험 콘텐츠는 복원하지 않는다. 위의 “3D 콘텐츠 완전 삭제” 절은 당시 정리 이력으로 보존하되, 전역 금지 규칙은 이번 정정으로 폐기한다.
- `tests/no_3d_content_smoke.gd`는 삭제하고 `tests/2d_identity_smoke.gd`로 대체했다. 새 검사는 핵심 게임플레이 경로의 승인되지 않은 3D 의존성만 차단한다.

### 참여 에이전트와 결론

- 기술: 640×360 `SubViewport`의 고정 직교 3D 배경을 `ViewportTexture`로 합성하고 캐릭터는 바깥 2D 레이어에 두는 방식을 추천했다.
- 비주얼: 작은 3D 필드는 조건부 찬성이나, 실루엣·픽셀 정합·폴백 중 하나라도 실패하면 전체 2D 컷인을 채택하도록 제안했다.
- 프로덕션: 전체 2D 컷인을 기준선이자 필수 폴백으로 먼저 완성하고 M4.1 격리 A/B 스파이크로 비교하도록 결정했다.
- combat: 2D/하이브리드 연출이 공유하는 결정론적 결과 payload와 비살상 피해 resolver를 구현했다.
- 2d-art/UI: 지휘관 1명+병사 8명씩, 대각 쿼터뷰, 1.42초 준비→충돌→후퇴 기준선을 구현했다.
- core/hybrid: 3D 지면·프롭만 포함하는 격리 미니 필드와 강제 실패 2D 폴백을 구현했다.

## Godot Visual Studio JIT 디버거 팝업 재발 방지

### 증상과 원인 판정

- Windows에서 `Godot_v4.7.1-stable_win64.exe`의 처리되지 않은 win32 예외를 알리는 Visual Studio Just-In-Time Debugger 창이 반복 표시됐다.
- 동일 프로젝트·동일 테스트는 정상 사용자 계정 `UNTIL-DESKTOP\until`에서 종료 코드 0으로 통과했다.
- 제한된 Codex Windows 샌드박스 계정에서 Godot을 직접 실행했을 때만 엔진이 `signal 11` 네이티브 충돌을 일으켰다.
- 여러 에이전트가 Godot 테스트를 동시에 시작해 충돌 프로세스와 JIT 창이 여러 개 생성된 것이 반복 표시의 직접 원인이었다.
- 현재 Godot, Visual Studio JIT, Windows Error Reporting 잔류 프로세스와 Godot CrashDump는 0개다.
- 판정: 프로젝트 GDScript 런타임 결함이 아니라 **제한 샌드박스에서의 Godot 네이티브 실행 및 병렬 검증 문제**다.

### 해결

- `tools/run_godot_check.ps1`를 추가했다.
  - `CodexSandboxOffline` 계정에서는 Godot을 시작하지 않고 코드 `86`으로 중단한다.
  - 기존 Godot 프로세스 또는 검증 잠금이 있으면 코드 `87`로 중단한다.
  - `editor`, `script`, `project` 모드를 한 번에 하나씩 실행하고 실제 Godot 종료 코드를 전달한다.
- `AGENTS.md`, `jumong-game-director`, `jumong-verify`, `docs/development.md`에 다음 규칙을 동기화했다.
  - Godot 검증은 총괄/검증 담당 단일 소유자가 정상 사용자 환경에서 순차 수행한다.
  - 서브에이전트는 Godot을 직접 병렬 실행하지 않는다.
  - Windows JIT 디버깅을 전역으로 끊어 충돌을 숨기지 않는다.

### 재검증

|---|---:|---|---|
| 제한 샌드박스 runner 방어 | 86 | 통과 | Godot 프로세스를 만들지 않고 실행 거부 |
| runner `editor` | 0 | 통과 | 파일 재탐색·전역 클래스 등록, 잔류 프로세스 없음 |

| 종료 후 프로세스 확인 | 0개 | 통과 | Godot/JIT/WerFault 잔류 없음 |

- 검증 판정: **통과 — 프로젝트는 정상 실행되며 동일 원인의 새 JIT 팝업 생성 경로를 차단했다.** 이미 떠 있던 과거 팝업은 확인 버튼으로 한 번 닫으면 된다.
