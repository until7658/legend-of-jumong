# 아트·맵·2D 캐릭터 통합 제작 파이프라인

## 목적과 적용 범위

이 문서는 주몽신화전기의 캐릭터 원화, 2D 캐릭터, 애니메이션, 타일, 맵과 Godot 통합이 서로 다른 독립 작업에서 같은 기준으로 이어지도록 하는 제작 계약이다. 모든 관련 작업은 시작 전에 이 문서와 아래의 기준 문서를 읽고, 각 단계의 승인 게이트를 통과한 산출물만 다음 단계의 입력으로 사용한다.

적용 영역은 `jumong-character`, `jumong-2d-art`, `jumong-level-2d`이며, 서사상 연령·세력·장면 요구는 `jumong-narrative`, 실제 리소스 연결은 담당 Godot 에이전트, 실행 판정은 `jumong-verify`가 소유한다.

## 현재 기준선과 즉시 조치

### P0 — 해결 전 후속 제작 중지

1. **주몽 연령 불일치**
   - `data/characters/main_cast.json`과 시나리오는 제1장 17세다.
   - `data/characters/jumong.json`은 탈출 시점 19세다.
   - PM이 공식 연령을 확정하고 두 데이터의 의미를 분리하거나 동기화하기 전에는 연령에 영향을 받는 신규 원화·체격·얼굴 변형 제작을 중지한다.
2. **골든 스타일 샘플 미승인**
   - 캐릭터는 선명한 윤곽선과 셀 채색을 기준으로 하지만, 맵 타일은 생성형 회화 질감과 정사영 재질 표현이 강하다.
   - 캐릭터 1종, 자연물 1종, 지면 1종을 같은 화면에 합성한 골든 샘플이 승인되기 전에는 신규 캐릭터 일괄 생성과 신규 지역 타일 일괄 생성을 중지한다.
3. **주몽 2D 원본 계보 불명확**
   - 애니메이션 사양은 `cartoon_v1/jumong.png`를 기준으로 하나 최신 주요 인물 시안은 `main_cast_v1/jumong.png`에도 존재한다.
   - PM이 어느 원화를 정본으로 지정하기 전에는 기존 프레임을 최종본으로 승격하거나 추가 동작을 제작하지 않는다.
4. **타일 연결성 미검증**
   - 타일 카탈로그 상태가 `needs_visual_peering_validation`이다.
   - Terrain Set과 반복 배치 화면 검증 전에는 프리뷰 맵을 최종 레벨로 승격하거나 충돌·전투 배치를 확정하지 않는다.

### P1 — 다음 마일스톤 전 보완

- 주몽 정면·측면·후면 턴어라운드, 활과 활집 분리 시트, 의상 겹침 구조도가 없다.
- `jumong_field_v1.png`는 1536×1024, `jumong_combat_v1.png`는 1252×1252여서 일관된 시트 격자 계약이 없다.
- 개별 프레임은 384×384 규격이지만 `idle` 전용 프레임 없이 `walk_front_00`을 재사용한다.
- 일부 PNG와 `.import` 짝이 불완전해 재탐색 결과를 기준으로 누락 여부를 재확인해야 한다.
- 맵 프리뷰는 대사·이동·충돌·전투가 없는 공간 확인용이며, 카메라 프레이밍과 캐릭터 화면 점유율 합성 증거가 없다.
- 타일 원본, 정규화본, Terrain Set 채택본의 상태를 파일명만으로 구분하기 어렵다.
- 주요 인물 17종은 콘셉트 시안일 뿐 최종 원화 승인 상태와 수정 이력이 개별 파일에 연결되어 있지 않다.

### P2 — 제작 안정화 단계 보완

- 초상화 표정 시트, 대화 UI 실제 크기 샘플, 무기·소품 공통 크기표가 필요하다.
- 색각 접근성을 포함한 세력색 검증표와 장면별 조명 변형표가 필요하다.
- Steam용 최종 AI 콘텐츠 공개 정책 및 상업 사용 검토 체크리스트가 필요하다.
- 맵 오브젝트의 2D 레이어 단계, 가림 페이드, 접지 그림자와 CanvasItem 조명 규격이 필요하다.

## 정본 우선순위

충돌 시 다음 순서로 판단한다.

1. 사용자가 해당 마일스톤에서 승인한 결정 기록
2. PM이 발행한 마일스톤 기준선과 이 문서
3. `docs/narrative/`의 승인된 제작 대본
4. `docs/characters/full_character_design_bible.md`
5. `data/characters/main_cast.json`
6. `docs/art/jumong_visual_guide.md`
7. 캐릭터·애니메이션·타일 개별 사양
8. 기존 이미지와 Godot 프리뷰

낮은 순위의 산출물이 높은 순위와 다르면 임의로 고치지 않고 PM에게 차이 보고를 올린다.

| 역할 | 필수 입력 | 소유 출력 | 승인 게이트 |
| --- | --- | --- | --- |
| PM·아트 디렉션 | 제작 대본, 캐릭터 바이블, 현행 애셋 | 스타일 기준선, 골든 샘플 선택, 작업 순서 | 사용자 방향 승인 |
| 캐릭터 원화 | 캐릭터 바이블, 세력 팔레트, 장면 요구 | 콘셉트, 턴어라운드, 표정, 장비 분해 | 외형·복식·연령·실루엣 승인 |
| 2D 캐릭터 | 승인 턴어라운드, 애니메이션 목록, 피벗 계약 | 프레임 PNG, 시트, 이벤트 프레임표 | 접지·방향·비례·무기 일관성 승인 |
| 애니메이션 | 승인 스프라이트 베이스, 전투 이벤트 타이밍 | 동작별 프레임, 루프·FPS·이벤트 태그 | 검수 씬 화면 승인 |
| 타일 아트 | 골든 환경 샘플, 장소 명도표, 셀 규격 | 콘셉트 시트, 정규화 아틀라스, 오브젝트 | 반복성·캐릭터 가독성 승인 |
| 레벨·맵 | 승인 Terrain Set, 시나리오 동선, 카메라 요구 | 맵 프리뷰, 충돌, 조우·컷신 마커 | 플레이 경로·가독성 승인 |
| Godot 통합 | 승인된 최종 후보, 카탈로그, 이벤트표 | `.tres`, `.tscn`, import 설정 | 헤드리스 로드 후 verifier 인계 |
| verifier | 승인 기준, 변경 목록, 실행 대상 | 화면·로그·종료 코드 기반 판정 | 사용자 확인 전 최종 완료 금지 |

한 역할은 다른 역할 소유 파일을 직접 수정하지 않는다. 수정이 필요하면 요청서를 PM에게 전달하고 소유자가 순차 반영한다.

## 파일과 폴더 계약

### 상태별 폴더

- 탐색 원화: `assets/concept/<set_id>/`
- 승인 턴어라운드: `assets/characters/<character_id>/turnaround/<version>/`
- 제작 프레임: `assets/characters/<character_id>/frames_<version>/`
- 승인 스프라이트 시트: `assets/characters/<character_id>/sheets/<version>/`
- 타일 콘셉트: `assets/tiles/concept/<region_id>/<version>/`
- 정규화 타일: `assets/tiles/normalized/<region_id>/<version>/`
- Godot TileSet: `resources/tilesets/<region_id>/<version>/`
- 맵 씬: `scenes/maps/<chapter_id>/<map_id>_<version>.tscn`

기존 경로는 이관 계획 전까지 보존한다. 새 작업부터 계약을 적용하며 대량 이동은 별도 승인 없이는 금지한다.

### 명명 규칙

- ID는 영문 소문자 `snake_case`만 사용한다.
- 버전은 작업본 `wip_v001`, 승인 후보 `rc_v001`, 승인본 `gold_v001`로 구분한다.
- 캐릭터 프레임: `<character_id>_<animation_id>_<direction>_<frame:02>.png`
- 턴어라운드: `<character_id>_turnaround_<view>_<version>.png`
- 표정: `<character_id>_portrait_<emotion>_<version>.png`
- 타일 아틀라스: `<region_id>_<layer>_<set_name>_<version>.png`
- 맵: `<chapter_id>_<location_id>_<purpose>_<version>.tscn`
- `final`, `new`, `latest`, `fix`는 파일명에 사용하지 않는다.

승인본은 덮어쓰지 않는다. 새 수정은 다음 버전으로 만들고 PM 승인 후 참조 경로를 변경한다.

## 캐릭터 제작 기술 규격

### 원화와 턴어라운드

- 콘셉트 원화: 1024×1536 이상, 인물 전체가 잘리지 않은 PNG.
- 턴어라운드: 정면·좌측·후면 필수, 필요 시 우측 추가. 동일한 머리 꼭대기와 지면선을 공유한다.
- 표정: 핵심 인물은 기본·기쁨·경계·분노·슬픔·결의 6종.
- 장비 분해: 활, 활집, 화살통, 칼, 외투와 긴 천은 몸체와 분리된 투명 레이어로 제공한다.
- 모든 원화에는 팔레트 칩, 키·무기 상대 길이, 금지 요소 확인표를 동봉한다.

### 2D 프레임

- 현재 프로토타입 기준 캔버스: 프레임당 384×384 투명 PNG.
- 피벗: `x=192`, 접지 `y=340`. 모든 방향과 동작에서 고정한다.
- 안전 여백: 상단·좌우 최소 16px. 활과 천이 넘으면 전체 프로젝트 규격을 일괄 확대해야 하며 개별 프레임만 확대하지 않는다.
- 방향 ID: `front`, `back`, `left`, `right`. 좌우 반전은 복식·활집 비대칭이 보이지 않는 경우에만 허용한다.
- 필드 이동: 기본 4프레임, 7 FPS, 루프.
- 전투 대기: 기본 4프레임, 5 FPS, 루프.
- 액션: 기본 4프레임 이상. 사양표의 FPS·루프·게임 이벤트 프레임을 명시한다.
- 접지 오차: 발 접점 ±2px, 머리 높이의 비의도적 흔들림 ±3px 이내.
- 각 동작은 `silhouette`, `grounding`, `costume`, `weapon`, `timing`, `event_frame` 항목으로 검수한다.

스프라이트 시트는 모든 셀이 같은 크기여야 하며 열은 프레임, 행은 방향 또는 동작으로 고정한다. `jumong_field_v1.png`와 `jumong_combat_v1.png`는 현 상태로는 서로 다른 격자이므로 참고본으로만 취급한다.

## 캐릭터 파이프라인

```text
서사·캐릭터 데이터 잠금
→ 콘셉트 원화 2안
→ PM 스타일·설정 검토
→ 사용자 1안 승인
→ 정면·측면·후면 턴어라운드
→ 장비·팔레트 분해
→ 골든 정지 스프라이트 1프레임
→ 맵 합성 가독성 검사
→ 4방향 이동
→ 전투·장면별 애니메이션
→ 프레임 이벤트표
→ Godot 검수 씬
→ verifier 판정
```

앞 단계 승인 전 다음 단계를 일괄 생산하지 않는다. 생성형 원화를 자동 분리한 프레임은 프로토타입이며, 접지·관절·활 형태를 사람이 정리하기 전 `gold`가 될 수 없다.

## 타일·맵 기술 규격

- 정규화 타일 아틀라스: 1024×1024, 4×4, 셀 256×256.
- 좌표 기준: 정사영. 카메라 각도와 광원 방향을 지역 내에서 고정한다.
- Ground와 Path는 불투명, 오브젝트는 투명 PNG.
- 오브젝트 피벗: 나무·선돌은 하단 중앙, 넓은 바위·통나무는 하단 접지 실루엣 중심, 소형 장식은 셀 중앙.
- 충돌: 나무는 줄기 하단, 바위는 실제 접지보다 작은 영역, 절벽 전면만 차단. 수관이나 전체 이미지에 충돌을 두지 않는다.
- Y 정렬 기준은 오브젝트 하단 접지점이다.
- 캐릭터의 화면상 키, 윤곽선 굵기와 지면 디테일 주파수를 골든 합성 샘플로 맞춘다.
- 타일에 인물, 대형 오브젝트, 그림자 방향이 다른 요소를 합치지 않는다.

## 타일·맵 파이프라인

```text
장소·동선·카메라 요구
→ 환경 무드·팔레트 콘셉트
→ 캐릭터 포함 골든 합성 샘플
→ 타일 콘셉트 분리(Ground/Path/Elevation/Detail/Object)
→ 1024×1024·256 셀 정규화
→ 반대편 경계와 3×3 반복 검사
→ Terrain Set peering 규칙
→ 충돌·Y 정렬·가림 설정
→ 맵 프리뷰
→ 실제 이동·카메라·전투 가독성 검사
→ 장면 씬 승격
→ verifier 판정
```

Terrain Set 승인 증거에는 각 재질의 직선, 내·외곽 모서리, 고립 패치, 경사·계단과 물가 연결을 실제 자동 페인팅한 화면이 포함되어야 한다.

## 스타일 합치기와 골든 샘플

골든 샘플은 다음 세 묶음으로 먼저 만든다.

1. **캐릭터 골든 샘플:** 17세 주몽 정면 전신, 필드 정지 1프레임, 활과 활집.
2. **환경 골든 샘플:** 풀·흙 경계 3×3 반복, 나무 1종, 바위 1종, 물가 1종.
3. **통합 골든 샘플:** 제1장 훈련장 한 화면에 주몽·표적·지면·나무·바위를 놓은 1280×720 캡처.

승인 기준:

- 캐릭터 윤곽선이 배경의 가장 강한 경계보다 명확하다.
- 배경은 캐릭터보다 채도와 대비가 한 단계 낮다.
- 같은 광원 방향과 따뜻한 림라이트 규칙을 공유한다.
- 캐릭터 셀 음영과 맵의 면 분할 밀도가 이질적으로 보이지 않는다.
- 필드 크기에서 머리, 어깨, 무기와 세력색으로 인물을 식별할 수 있다.
- 생성형 격자, 흐린 외곽, 비일관 그림자와 사실적 사진 질감이 없다.

골든 샘플 승인 후 지역·인물 양산을 시작한다. 스타일 변경은 기존 전체를 즉시 다시 만들지 않고 골든 샘플만 먼저 `rc`로 교체해 비용을 산정한다.

## 재작업 기준

### 반드시 재작업

- 승인 정본과 나이·세력색·복식·무기가 다름.
- 턴어라운드에서 얼굴, 머리, 의상 구조가 방향마다 변함.
- 프레임 사이 활 길이·손가락·관절·피벗이 흔들림.
- 타일 반대편 경계가 연결되지 않거나 반복 격자가 눈에 띔.
- 캐릭터가 맵 명도·채도·디테일에 묻힘.
- 저작권·AI 기록이 누락되거나 특정 작품을 직접 모사함.

### 조건부 허용

- 컷신용 원화와 필드 스프라이트의 세부 묘사 차이.
- 게임 해상도에서 보이지 않는 작은 장식 단순화.
- 프로토타입에 한정된 임시 색상·충돌. 단, 파일과 문서에 `wip` 표시가 있어야 한다.

## 파일 소유권과 충돌 방지

- PM·시나리오: `docs/narrative/`, 마일스톤 기준선.
- 캐릭터 기획: `docs/characters/`, `data/characters/`.
- 아트 디렉터·원화: `docs/art/jumong_visual_guide.md`, `assets/concept/`.
- 2D 담당: `docs/art/*_2d_animation_spec.md`, `assets/characters/`.
- 맵 아트: `docs/art/tilemap_asset_spec.md`, `assets/tiles/`, `assets/objects/`, `data/tilesets/`.
- 레벨 담당: `docs/maps/`, `scenes/maps/`.
- Godot 캐릭터 통합: `scenes/characters/`, `scripts/characters/`, `resources/characters/`.
- AI 기록: `docs/assets/ai_asset_log.md`는 PM이 작업 종료 보고를 받은 뒤 순차 반영한다. 여러 작업이 동시에 수정하지 않는다.

작업 시작 시 담당자는 PM에게 `LOCK: <path>`를 보고한다. PM은 소유자·시작 시각·예상 종료를 기록한다. 같은 파일을 필요로 하는 작업은 병렬 수행하지 않는다. 읽기 참조는 가능하지만 잠긴 파일의 수정은 금지한다. 작업 종료 시 변경 파일과 해시 또는 크기, 검증 결과를 보고하고 PM이 잠금을 해제한다.

## 버전과 승인 상태

- `wip`: 제작 중, 게임 참조 금지.
- `rc`: 승인 후보, 검수 씬·합성본에서만 참조.
- `gold`: 사용자와 PM이 승인한 현재 기준본.
- `deprecated`: 비교·이력용, 신규 참조 금지.

버전은 의미 변경 시 증가한다. 단순 import 재생성은 이미지 버전을 올리지 않되 로그에 기록한다. `.tscn`과 데이터가 특정 애셋을 참조하기 시작하면 그 애셋은 덮어쓰지 않고 새 버전 경로로 교체한다.

## AI 애셋 기록 시점

1. 생성 직후: 도구, 날짜, 프롬프트 목적, 참고 자료, 원본 위치 기록.
2. 프로젝트 복사·정규화 직후: 파일 경로, 크기, 후처리, 분류 기록.
3. `rc` 제출 전: 사용 위치, 수정자, 유사성·복식·인체 검토 상태 기록.
4. `gold` 승인 시: 승인자, 사용 씬, 상업 사용 검토, 추가 수작업 여부 기록.
5. Steam 제출 전: 최신 공식 AI 콘텐츠 공개 정책으로 재검토.

`docs/assets/ai_asset_log.md`에는 최종 기록만 순차 병합하고, 각 작업은 먼저 자기 산출물 폴더에 `<asset_id>_generation_note.md` 또는 PM 보고 블록을 제공한다.
