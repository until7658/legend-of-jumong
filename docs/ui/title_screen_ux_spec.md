# 타이틀 화면 UI/UX 명세

검수용 와이어프레임: [`assets/ui/title/title_screen_wireframe.svg`](../../assets/ui/title/title_screen_wireframe.svg)

## 1. 목적과 범위

- 대상 화면: 게임 실행 직후 표시되는 정식 타이틀 화면
- 기준 해상도: 1280×720, 16:9
- 역할: 작품의 첫인상을 만들고 `새 게임`을 프롤로그 「강에서 온 여인」으로 연결한다.
- 서사 원칙: 강, 안개, 비가 갠 새벽빛으로 작품의 시작을 암시하되 햇빛이나 자연현상을 초자연적 징표로 확정하지 않는다.
- 비주얼 원칙: 동부여의 청흑색과 회청색을 바탕으로, 적동색은 포커스와 확인 동작에 제한한다. 삼족오·태양 문장·왕관·금색 판금 이미지는 사용하지 않는다.
- 이 문서는 레이아웃, 상호작용, 모션과 개발 전달 기준만 정의한다. 키비주얼, 로고 원화, 음원과 Godot 구현은 별도 작업이다.

## 2. 정보 위계

1. 작품 로고 `주몽신화전기`
2. 부제 `한 사람의 삶은 역사가 되고, 역사는 마침내 신화가 되었다.`
3. 주 행동 `새 게임`
4. 보조 행동 `이어하기`, `설정`, `종료`
5. 빌드 버전과 저작권·공지 정보

첫 실행에서는 저장 데이터가 없으므로 `새 게임`이 최초 포커스를 가진다. 저장 데이터가 생긴 정식 빌드에서는 `이어하기`를 활성화하되 최초 포커스 정책은 사용자 테스트 후 결정한다.

## 3. 1280×720 와이어프레임

좌표 원점은 화면 좌상단이며 모든 수치는 px이다. 배경은 화면 전체를 덮고, UI는 안전영역 안에 배치한다.

```text
0,0 ┌──────────────────────────────────────────────────────────────┐ 1280
    │  안전영역 48                                                │
    │                                                              │
    │                         [로고]                               │
    │                    x=300 y=126 w=680 h=96                    │
    │                         [부제]                               │
    │                    x=300 y=232 w=680 h=28                    │
    │                                                              │
    │                    ┌────────────────┐                        │
    │                    │    새 게임     │ y=356                  │
    │                    │    이어하기    │ y=412                  │
    │                    │      설정      │ y=468                  │
    │                    │      종료      │ y=524                  │
    │                    └────────────────┘                        │
    │                                                              │
    │ [비활성 사유/도움말]                     [빌드 버전]          │
    │ x=48 y=650 w=680 h=22                    x=1032 y=650 w=200   │
    │  안전영역 48                                                │
720 └──────────────────────────────────────────────────────────────┘
```

### 3.1 안전영역과 정렬

- 기본 안전영역: 좌우 48, 상하 40. 핵심 텍스트와 상호작용 요소는 이 안에 둔다.
- 방송·TV 오버스캔 대응 권장영역: 좌우 64, 상하 48. 게임패드가 주요 입력인 플랫폼에서는 이 값을 우선한다.
- 로고와 메뉴는 화면 중앙축 `x=640`에 맞춘다.
- 키비주얼의 주 피사체는 중앙 UI 뒤가 아니라 좌우 1/3 영역에 두어 텍스트 대비를 확보한다.
- 로고 영역: `x=300, y=126, w=680, h=96`. 실제 로고는 영역 안에서 원본 비율을 유지한다.
- 부제 영역: `x=300, y=232, w=680, h=28`. 로고 하단과 최소 10 간격.
- 메뉴 영역: `x=476, y=356, w=328, h=212`.
- 버튼: `w=328, h=48`, 버튼 사이 세로 간격 8. 시각적 텍스트 정렬은 중앙.
- 하단 도움말과 빌드 버전: 기준선 `y=650`, 글자 높이 22. 빌드 버전은 우측 정렬.

## 4. 로고·부제·메뉴 표현

### 4.1 로고

- 한글 판독성을 최우선으로 한다. 1280×720 기준 실제 글자 높이는 64~78 범위다.
- 밝은 하늘 위에서도 읽히도록 1~2 px의 청흑색 외곽선과 부드러운 그림자를 사용한다.
- 질감은 먹, 마른 붓, 산화된 청동에서 착안하되 과도한 금속 광택이나 왕실 문장은 피한다.
- 로고가 이미지로 공급되지 않은 단계에서는 내장 한글 폰트의 64 px 텍스트를 플레이스홀더로 사용한다.

### 4.2 부제

- 18 px, Medium 또는 Semibold 권장. 한 줄 고정이며 말줄임표를 사용하지 않는다.
- 배경 대비가 부족할 때 글자 그림자보다 로고·부제 영역 뒤의 국소 암막 그라데이션을 우선한다.

### 4.3 메뉴

- 메뉴는 테두리 상자보다 텍스트와 가는 수평선, 좌측의 짧은 적동색 포커스 표식으로 구성한다.
- 실제 클릭·터치 영역은 시각 요소보다 작아지지 않으며 최소 높이 48을 유지한다.
- 버튼 레이블은 20 px. 모든 상태에서 글자 위치가 움직이지 않아야 한다.

## 5. 버튼 상태

| 상태 | 글자 | 배경/선 | 포커스 표식 | 모션 |
|---|---|---|---|---|
| Default | `#C6CDD2` | 투명, 하단선 `#71808A` 28% | 없음 | 없음 |
| Hover | `#F1E8CF` | 청회색 `#263844` 42% | 적동색 2 px, 높이 20 | 100 ms 색 전환 |
| Focus | `#FFF2CF` | 청회색 `#263844` 58% | 적동색 3 px, 높이 28 + 외곽선 | 120 ms 등장, 미세한 수면광 1회 |
| Pressed | `#FFF7E5` | 적동색 `#8E4D36` 38% | 3 px 유지 | 70 ms 밝아짐 후 90 ms 복귀 |
| Disabled | `#75808A` | 투명, 하단선 `#56636B` 18% | 없음 | 없음 |

- Hover는 마우스에만 적용한다. 키보드·게임패드의 현재 위치는 항상 Focus로 표현한다.
- Focus는 색만으로 전달하지 않는다. 좌측 표식, 2 px 외곽선 또는 밑줄 중 최소 두 가지를 함께 사용한다.
- Disabled는 포커스 목록에는 남겨 비활성 사유를 확인할 수 있게 하되 확인 입력은 실행하지 않는다. Godot의 단순 `disabled=true`가 포커스를 차단한다면 별도 설명 가능한 메뉴 항목 래퍼로 구현한다.

## 6. 입력과 포커스 흐름

### 6.1 공통 순서

```text
새 게임 → 이어하기 → 설정 → 종료 → 새 게임
새 게임 ← 이어하기 ← 설정 ← 종료 ← 새 게임
```

- 위/아래, 방향패드, 좌측 스틱은 위 순서로 순환한다.
- 최초 포커스는 `새 게임`이다.
- 마우스를 움직이면 포인터가 올라간 항목을 Hover로 표시하되, 마지막 키보드·게임패드 Focus는 내부적으로 기억한다.
- 마우스 클릭 시 해당 항목에 Focus를 먼저 부여하고 같은 클릭에서 확인한다.
- 입력 장치가 바뀌어도 포커스가 사라지지 않는다.

### 6.2 키보드

- 이동: `↑/↓`, 선택: `Enter` 또는 `Space`, 취소: `Esc`.
- 타이틀 루트에서 `Esc`는 즉시 종료하지 않고 종료 확인 대화상자를 연다.
- 종료 확인: `종료하시겠습니까?` / `아니요`(기본 포커스), `예`.

### 6.3 게임패드

- 이동: 방향패드/좌측 스틱, 선택: South(A/Cross), 취소: East(B/Circle).
- 스틱 반복 입력은 초기 300 ms 지연 후 110 ms 간격을 권장한다.
- 취소 입력은 종료 확인 대화상자를 열고, 대화상자에서 다시 취소하면 닫는다.

### 6.4 마우스

- 왼쪽 클릭으로 선택한다. 버튼 전체 328×48이 클릭 영역이다.
- 포인터가 화면 밖으로 나가도 기존 Focus는 유지한다.
- 클릭하지 않은 Hover만으로 선택음이나 설명 음성을 반복 재생하지 않는다.

## 7. 비활성 메뉴 설명

- `이어하기`: 저장 데이터가 없으면 비활성. 포커스/호버 시 하단 도움말에 `저장된 여정이 없습니다.` 표시.
- `설정`: 프로토타입에서 미구현이면 비활성. `설정 기능은 준비 중입니다.` 표시.
- 비활성 사유는 하단 좌측 `x=48, y=650`에 16 px로 표시하며, Focus 이동 후 100 ms 안에 나타난다.
- 사유 문구는 툴팁처럼 버튼을 가리거나 위치를 바꾸지 않는다.
- 정식 출시 버전에서 `설정`은 타이틀에서도 활성화해야 하며 접근성·음량·화면 옵션에 새 게임 전에 접근할 수 있어야 한다.

## 8. 새 게임에서 오프닝으로의 연결

1. 확인 입력 즉시 버튼을 Pressed로 160 ms 표현하고 중복 입력을 잠근다.
2. 메뉴와 부제가 300 ms 동안 먼저 사라진다. 로고는 150 ms 늦게 시작해 450 ms 동안 사라진다.
3. 배경의 환경음은 끊지 않고 유지하며 600 ms 동안 프롤로그의 강물 음향 레이어로 교차한다.
4. 화면은 500 ms 동안 암전한다. 총 전환 시간은 약 900 ms다.
5. 프롤로그 컷 1의 검은 화면과 강물 소리로 진입한다.
6. 프롤로그 로드가 실패하면 타이틀로 복귀하고 `오프닝을 불러오지 못했습니다.`를 하단에 표시한다.

현재 시험장으로 직접 진입하는 연결은 개발용 임시 경로다. 정식 수직 슬라이스에서는 `new_game_requested` 신호의 수신 대상만 프롤로그 씬 로더로 교체하고, 타이틀 UI가 게임 흐름 로직을 직접 소유하지 않게 한다.

## 9. 해상도와 화면비 대응

- 기준 UI 스케일: 1280×720에서 1.0. 1920×1080에서는 1.5를 기준으로 하되 폰트 래스터 품질을 유지한다.
- 최소 지원 해상도 권장: 1024×576. 이때 로고 54 px, 부제 16 px, 메뉴 버튼 44 px 높이까지 축소할 수 있다.
- 16:10: 수평 배치는 유지하고 여분의 세로 공간을 로고 위와 메뉴 아래에 반씩 분배한다.
- 21:9 이상: UI 폭은 1280 기준 중앙 16:9 컴포지션 안에 유지하고, 배경만 좌우로 확장한다. 키비주얼 크롭 안전선을 별도 납품한다.
- 4:3: 로고 폭을 화면의 78% 이하로 줄이고, 부제를 최대 2줄로 허용한다. 메뉴는 중앙 유지.
- 창 크기 변경 중 앵커와 컨테이너로 재배치하며 절대 픽셀 좌표는 기준 캔버스에서만 사용한다.
- 배경은 비율 유지 Cover 방식으로 채우고 중요한 피사체는 중앙 70% 안전구역에 둔다.

## 10. 접근성

- 일반 텍스트 대비는 WCAG AA 기준 4.5:1 이상, 큰 로고·버튼 텍스트는 3:1 이상을 목표로 한다.
- 포커스는 색상 외 표식과 윤곽으로 구분한다.
- 최소 상호작용 높이 48 px, 항목 간 8 px 간격을 유지한다.
- 화면 번쩍임은 사용하지 않는다. 밝기 변화는 초당 3회 미만이며 전체 화면의 급격한 백색 전환을 피한다.
- 모션 감소 옵션이 켜지면 안개·수면 이동을 정지 이미지로 바꾸고, 등장·전환은 150 ms 단순 페이드로 축소한다.
- 텍스트 크기 확대 125%에서도 메뉴가 겹치지 않도록 메뉴 영역 높이를 컨테이너 기반으로 확장한다.
- 선택음과 함께 시각 피드백을 제공한다. 정보 전달을 음향에만 의존하지 않는다.
- 한글 폰트는 `주몽`, 받침, 따옴표와 말줄임표를 포함한 전체 글리프를 확인하고, 저해상도에서 획 뭉침을 검사한다.

## 11. 등장·선택·전환 모션

### 11.1 최초 등장

| 시간 | 요소 | 동작 |
|---:|---|---|
| 0~400 ms | 배경 | 암전에서 새벽 강으로 페이드 인 |
| 250~850 ms | 로고 | 투명도 0→1, y +8→0, Ease Out Sine |
| 600~1000 ms | 부제 | 투명도 0→1 |
| 850~1250 ms | 메뉴 | 위에서 아래로 항목별 60 ms 시차 페이드 |
| 1150~1400 ms | 하단 정보 | 투명도 0→1 |

- 최초 입력은 메뉴가 나타난 850 ms부터 허용한다. 입력 시 남은 등장 모션을 즉시 완료할 수 있어야 한다.
- 배경 수면·안개 루프는 12~20초 주기로 느리게 움직이고 눈에 띄는 반복 경계를 피한다.

### 11.2 포커스와 확인

- 포커스 이동: 120 ms. 항목 자체의 좌우 위치는 바꾸지 않는다.
- 선택음은 포커스가 실제로 다른 항목으로 이동했을 때 1회만 재생한다.
- 확인: 160 ms Pressed 피드백 후 해당 행동을 수행한다.
- 비활성 확인: 100 ms의 낮은 음과 포커스 표식 1회 점멸. 화면 흔들림은 사용하지 않는다.

## 13. 현재 프로토타입과의 차이

| 항목 | 현재 프로토타입 | 정식 명세 |
|---|---|---|
| 구성 | 620×410 중앙 VBox | 안전영역 기반 로고·메뉴·하단 도움말 영역 분리 |
| 메뉴 폭/높이 | 280×42~44 | 328×48, 8 간격 |
| 상태 | 기본 Godot 버튼 스타일 | 5개 상태와 비색상 포커스 표식 정의 |
| 비활성 항목 | 포커스 불가, 설명 없음 | 포커스/호버로 사유 확인 가능 |
| 포커스 | 최초 새 게임만 지정, 일부 이웃만 설정 | 키보드·게임패드 상하 순환과 장치 전환 유지 |

| 해상도 | 전체 앵커 외 별도 규칙 없음 | 4:3~21:9, 최소 해상도와 크롭 안전선 정의 |

| 배경 | 셰이더 기반 새벽 강 | 프로토타입 유지 가능, 이후 레이어 키비주얼로 교체 가능 |

## 14. 구현 우선순위

### P0 — 기능·접근성 기준선

1. 메뉴 버튼 328×48과 8 간격 적용
2. 완전한 상하 순환 포커스와 입력 장치 전환 처리
3. 비활성 메뉴 사유를 표시하는 하단 도움말 추가
4. `Esc`/취소의 종료 확인 대화상자 구현
5. `새 게임`을 프롤로그 씬 로더 신호에 연결
6. 1024×576, 1280×720, 1920×1080과 21:9에서 레이아웃 검사

### P1 — 정식 시각 상태와 모션

1. 디자인 토큰 기반 Theme/StyleBox 정의
2. Default/Hover/Focus/Pressed/Disabled 상태 구현
3. 최초 등장과 새 게임 전환 모션 적용
4. 포커스·확인·비활성 사운드 UX 적용
5. 모션 감소 분기 구현

### P2 — 최종 애셋 통합

1. 승인된 타이틀 로고 이미지 교체
2. 레이어 분리된 강·안개·하늘 키비주얼 적용
3. 화면비별 크롭과 명도 대비 보정
4. Steam 빌드 정보, 저작권과 현지화 문구 적용

## 15. 검수 체크리스트

- 최초 포커스가 `새 게임`이며 시각적으로 명확한가
- 모든 입력 장치에서 메뉴를 순환하고 선택할 수 있는가
- 비활성 항목의 이유를 읽을 수 있는가
- 취소 입력이 실수로 게임을 즉시 종료하지 않는가
- 1280×720에서 로고, 메뉴, 도움말과 버전이 안전영역 안에 있는가
- 4:3과 21:9에서 UI가 잘리거나 키비주얼 핵심부를 가리지 않는가
- 포커스를 색상 없이도 식별할 수 있는가
- 모션 감소 상태에서 모든 기능이 유지되는가
- `새 게임` 선택 후 중복 입력이 차단되고 프롤로그 컷 1로 이어지는가
- 로드 실패 시 타이틀로 안전하게 복귀하는가

## 16. 파일 소유권과 후속 인계

- UI/UX: 본 문서의 레이아웃·입력·상태·접근성 기준 소유
- 로고/키비주얼: 일러스트·로고 담당이 별도 애셋으로 제작
- Godot 구현: UI 개발 담당이 씬·스크립트·Theme 리소스에 반영
- 서사 연결: 프롤로그 제작 대본과 장면 로더가 `새 게임` 흐름을 소유
- 검증: 제작 파일을 수정하지 않는 별도 검증 단계에서 화면·입력·로그로 판정

현재 프로토타입의 셰이더 배경은 P0~P1 동안 플레이스홀더로 유지할 수 있다. 최종 키비주얼이 들어와도 배치와 입력 구조는 변경하지 않는 것을 원칙으로 한다.
